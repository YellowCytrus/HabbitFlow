# Обработка ошибок

## Текущая модель ошибок

- Основной механизм: `HTTPException(status_code, detail)`.
- Ошибки валидации: стандартные ответы 422 от FastAPI/Pydantic.
- Нет глобального формата custom-exception с устойчивыми машиночитаемыми кодами.

## Типовые категории ошибок

- Authentication: 401 для невалидного/просроченного/отсутствующего токена.
- Authorization/policy: 403 для отключенного аккаунта или тарифных ограничений.
- Domain constraints: 400 для неверной даты лога/статуса/конфликтов email.
- Missing resources: 404 для отсутствующей привычки/неизвестного OAuth-провайдера.
- Operational failures: 503 readiness при недоступной БД.

## Обработка внешних сбоев

- Отправка email в фоновых сценариях ловит и логирует исключения.
- Сбои внешних OAuth API приводят к frontend-редиректу с ошибкой.
- SMTP-логика содержит fallback по протоколу и fallback на Resend API.

## Риски и направления улучшения

- Строковые значения `detail` нестабильны как frontend-контракт.
- Мало correlation-метаданных в теле ошибок (request ID есть только в заголовке).
- Разный стиль exception-handling между роутерами может давать непоследовательное клиентское поведение.
