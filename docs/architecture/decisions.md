# Архитектурные решения

## Слоистый монолит вместо микросервисов

**Решение**: оставить единый backend-сервис с явными внутренними модулями.

**Причины**:
- Ниже операционная сложность для MVP.
- Быстрее итерации по auth, привычкам и напоминаниям.
- Единая транзакционная граница в одной базе данных.

**Компромисс**: меньше изоляции; границы модулей логические, а не деплойные.

## Прямой доступ к ORM из роутеров

**Решение**: роутеры могут напрямую читать/изменять данные через SQLAlchemy-сессию.

**Причины**:
- Меньше прослоек и ниже церемониальность для CRUD-сценариев.
- Проще вход в проект для небольшой команды.

**Компромисс**:
- Сложнее централизовать инварианты.
- Файлы эндпоинтов сильнее связаны с persistence-слоем.

## Гибридная доставка напоминаний

**Решение**: backend рассчитывает due-напоминания, frontend опрашивает `/api/v1/notifications/due` и показывает браузерные уведомления.

**Причины**:
- Нет немедленной необходимости в очереди/worker/push-канале.
- Работает в рамках уже аутентифицированной браузерной сессии.

**Компромисс**:
- Надежность зависит от активности вкладки и разрешений браузера.
- Worker-логика есть в коде, но не является основным production-путем.

## Стратегия синхронизации миграций

**Решение**: использовать Alembic с идемпотентной логикой синхронизации схемы.

**Причины**:
- Практично для быстрого выравнивания дрейфа между моделями и схемой.

**Компромисс**:
- Гарантии downgrade слабее.
- Нужна более строгая дисциплина релизов и миграций в production.

## Известные ограничения и технический долг

- Нет выделенного runtime-worker для обработки/self-healing уведомлений в текущем compose.
- Контракты ошибок не стандартизованы (нет единого формата problem details).
- Разделение router/service неоднородно между доменами.
- Поведение таймзон разделено между backend-настройками и frontend-автосинхронизацией.
