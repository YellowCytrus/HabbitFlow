# Стратегия миграций

Конфигурация миграций находится в `backend/alembic`. Выполнение миграций в рантайме входит в старт контейнера.

## Политика в текущем коде

- Есть начальные и инкрементальные ревизии Alembic для auth, уведомлений и изменений схемы.
- Последняя ревизия (`009_models_sync`) выполняет идемпотентную синхронизацию по metadata SQLAlchemy.
- Downgrade для последней sync-миграции намеренно оставлен no-op.

## Почему так сделано

- Проект быстро эволюционировал, поэтому приоритетом стала коррекция дрейфа схемы.
- Подход через metadata снижает трение при добавлении новых колонок/таблиц.

## Операционные последствия

- Гарантии rollback слабее, чем при строго ручных forward/backward-миграциях.
- В production миграции стоит рассматривать как forward-only, если обратный путь не проверен в staging.
- Backfill по default/nullable в sync-миграции может скрывать рассинхрон между моделью и дизайном.

## Рекомендуемые практики

- По возможности делать явные миграции для новых изменений схемы.
- Добавить smoke-тесты миграций в CI для критичных релизов.
- Фиксировать необратимые ревизии в release notes.
