# Стратегия тестирования

Тестирование в репозитории ориентировано на backend и содержит много интеграционных сценариев.

## Инструменты

- Раннер: `pytest`.
- Тестовая БД: PostgreSQL через testcontainers по умолчанию или внешний инстанс через `TEST_DATABASE_URL`.
- Скрипт запуска: `run_tests.sh` с синхронизацией окружения через `uv`.

## Зоны покрытия

- Поведение API-эндпоинтов (auth, habits, profile, calendar, notifications).
- Security-сценарии (токены и авторизация).
- Математика расписания уведомлений.
- Lock/reschedule-поведение notification worker.
- Восстановление уведомлений в self-healing логике.

## Сильные стороны

- Тесты проверяют реальное поведение ORM + PostgreSQL (включая JSONB и ограничения).
- Ключевая логика расписания и lock-механизма покрыта прямыми тестами.
- API-тесты проверяют лимиты тарифа, владение ресурсами и граничные случаи валидации.

## Пробелы и риски

- В репозитории нет автоматизированного frontend test-suite.
- Нет e2e smoke-тестов деплоя на уровне CI-документации.
- Нет contract-тестов для единой схемы ошибок, поскольку схема не централизована.
