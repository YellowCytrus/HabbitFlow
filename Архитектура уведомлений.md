
# Уведомления

1. Создаём таблицу для уведомлений `Notifications`

храним `notification_id`, `user_id`, `goal_datetime` (храним в UTC), `payload`, `is_active`, `FK на NotificationSettingsOut`, `last_check_at` (храним в UTC), `is_processing`, `user_timezone`.

Поле `payload` получаем заранее перед записью в `Notifications` (основываясь на `NotificationSettingsOut`)

Поле `is_processing` нужно для избежания Race Condition между Self-healing скриптом и воркером.

`NotificationSettingsOut` нужен чтобы воркер получал актуальную информацию про тип Repetition (каждый день или неделю), чтобы вычислять `delay` для будущей таски, а так же кейс: Если пользователь изменит время уведомления в 12:00 на 15:00, вы просто обновляете SQL. Воркер в 12:00 увидит в SQL новое время и просто перепланирует задачу на 15:00, не отправляя пуш сейчас -- то есть проверка "а надо ли отправлять уведомление прямо сейчас?" обязательна для консистентности

Поле `is_active` нужно для индикации актуальности уведомления. Если оно `true`, то уведомление отправится в указанное время. Оно может стать `false` если по той или иной причине план на уведомление отменён (пользователь удалил привычку или отключил уведомления)

Получается уведомления будут вечно висеть активными 

2. Брокер (celery) -- очередь сообщений

В Celery (с Redis) лучше использовать ETA (Estimated Time of Arrival) или countdown. Вы один раз создаете задачу при создании/изменении настройки. Воркер сам пересоздает следующую задачу в конце своей работы. «Каждую минуту» ничего в очередь пушить не нужно.

Формат элемента очереди: notification_id, delay 

delay предварительно вычисляем по принципу `goal_datetime` - `current_datetime` 


3. Воркер

Когда время пришло, Брокер «будит» воркера и отдает ему notification_id

Воркер делает следующее:
- Идет в SQL и проверяет: «А активно ли еще это уведомление? Не удалил ли его пользователь за это время?».
- Если всё ок — берет данные подписки и отправляет Push.
- Записывает лог отправки в таблицу notification_logs.

Логика воркера:
1. Проснулся в 09:00 (когда его разбудил брокер)
2. При старте воркер должен делать UPDATE Notifications SET is_processing = 1 WHERE notification_id = X AND is_processing = 0. Если затронуто 0 строк – значит уже обрабатывается, выходим (избегаем гонки между двумя воркерами над одним уведомлением, мало ли)
3. Отправил пуш (если планы на пуш не изменились)
4. Вычислил время для следующего пуша (например, завтра в 09:00 или через неделю).
5. Закинул новую задачу в Celery.
6. Обновил goal_datetime в SQL на завтра (или на следующую неделю).
7. После обработки сбросить is_processing = 0 (или оставить 0 после обновления goal_datetime).
8. Обновил last_check_at на «сейчас».


!!!Тут внимание!!!
При вычислении «следующего завтрашнего 09:00» сначала переводи UTC в локальное время пользователя, добавляй 24 часа, и конвертируй обратно в UTC для записи в базу

4. Этап «Рекурсия»

После успешной отправки воркер сам рассчитывает время для следующего уведомления (например, +24 часа).
Он создает в брокере новую задачу с новым delay.
Таким образом, в очереди всегда висит только одна будущая задача для каждого уведомления. Это не забивает память Redis миллионами записей на год вперед.

5. Self-Healing

Self-healing нужен на случай, если Celery «забыл» задачу (например, упал Redis без сохранения данных или случился баг в коде рекурсии).

По сути это будет отдельный скрипт (Beat-задача в Celery), который запускается, например, раз в 5 минут.

Логика скрипта:
1. Делает запрос в SQL: SELECT * FROM Notifications WHERE is_active = true.
2. Для каждой записи проверяет два случая:
    1. Случай А (зависший воркер): если is_processing = 1 и last_check_at старше 10 минут — снимает блокировку (is_processing = 0).
    2. Случай Б (пропущенное уведомление): если goal_datetime уже в прошлом, is_processing = 0, а текущее время больше goal_datetime — значит рекурсия не сработала.
3. Действие: Скрипт перепланирует уведомление: вычисляет следующее goal_datetime (начиная с текущего момента), обновляет запись в БД и закидывает новую задачу в Celery. Если опоздание небольшое (например, до 1 часа), можно отправить уведомление сразу перед перепланированием.


7. На случай смены пользователем часового пояса

Скрипт который отзовёт у брокера все задачи на отправку пуш у данного пользователя, а потом просто сгенерирует новые
